Make the sidebar behave as a mobile drawer below `lg` and keep it identical to today on `lg+`. Three surgical changes:

Purpose: Satisfies LAYOUT-02 (drawer mechanics below `lg`) and LAYOUT-05 (safe-area respected on notch/home-indicator devices) without regressing the desktop layout.

    In `src/stores/workspaceStore.ts`, add a helper function near the top (above the `WorkspaceState` interface, alongside `getInitialTheme`):

    ```typescript
    function getInitialSidebarOpen(): boolean {
      if (typeof window === 'undefined') return true
      return window.matchMedia('(min-width: 1024px)').matches
    }
    ```

    The breakpoint `1024px` matches Tailwind's default `lg`. Below that → drawer closed on first paint. On desktop → sidebar visible as before.

    Then, in the store's `create()` return object, replace the line:


    **Change 2 — Focus-return-to-hamburger on drawer close in Sidebar.tsx:**

    In `src/components/Sidebar.tsx`:

    1. Add `useEffect, useRef` to the React import at the top of the file. The file currently has no React import (only lucide + internal). Add:

       ```tsx
       import { useEffect, useRef } from 'react'
       ```

       Place it as the FIRST import line.

    2. Inside the `Sidebar()` function body, after the existing `useWorkspaceStore` selector calls and before `handleSignOut`, add:

       ```tsx
       const prevSidebarOpen = useRef(sidebarOpen)
       useEffect(() => {
         const wasOpen = prevSidebarOpen.current
         prevSidebarOpen.current = sidebarOpen
         // Only return focus on the open → closed transition, and only when a hamburger exists (below lg).
         if (wasOpen && !sidebarOpen) {
           const hamburger = document.querySelector<HTMLButtonElement>('[data-sidebar-trigger="true"]')
           if (hamburger) hamburger.focus()
         }
       }, [sidebarOpen])
       ```

       This uses a DOM query keyed to a `data-sidebar-trigger` attribute (added in Task 3 of plan 04-03 on the Canvas hamburger button). Using a data attribute is intentional: the Sidebar component does not own the hamburger, so a DOM-level handoff is simpler than threading refs through `WorkspaceLayout`.

       Rationale: on mobile, focus would otherwise land nowhere after a tap-to-close, which fails keyboard-accessibility and screen-reader flows. Returning focus to the trigger is the standard dialog pattern (per CONTEXT "return focus to the hamburger trigger on close").

    Do NOT auto-focus any element when the drawer opens (user deferred that). Do NOT trap focus inside the drawer (not required for v1.1). Do NOT add ESC handling (user deferred).
