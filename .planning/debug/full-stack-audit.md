## Resolution

root_cause:
  PRIMARY: Supabase Edge Function `fal-proxy` hits WORKER_RESOURCE_LIMIT when forwarding requests to fal-ai/nano-banana-pro/edit. This causes: (a) "credits consumed but no image shown" (FAL ran, edge function died before returning), and (b) user-reported "num_images=2 returns 1" symptom is actually "one of the two parallel requests died" OR "the single request died and only the first of a fan-out survived" depending on mode.
  SECONDARY: supabase/migrations/001_initial_schema.sql CHECK constraint on `generations.mode` does NOT include 'video'. Every video save silently fails in persistGeneration.catch(...).
  TERTIARY: Posting multi-MB base64 data URLs directly to the edge function inflates request size and CPU cost; contributes to resource-limit hits.
  QUATERNARY: `falRequest` error parser misses FAL's `{code, message}` error envelope — users see opaque "HTTP 500" instead of "WORKER_RESOURCE_LIMIT".
  SEPARATE: The --no-verify-jwt workaround opens fal-proxy to anonymous abuse; must add in-function JWT verification.
